## Модель сетевого пути (эволюция карты)

> **Стратегия:** не углубляться в K8s-сущности (поды, деплойменты) — это решено Grafana/Lens/k9s.
> Вместо этого моделировать **логический путь** трафика: User → VPN → Corp → Interconnect → VPC → LB → Ingress → Service → Managed DB.
> Каждый хоп — нода на карте со своим статусом из blackbox-проб или облачного API.
> K8s-кластер — «чёрная коробка»: видим ingress и сервисы, но не отдельные поды.

### [ ] Фаза 1: Семантический тип узла (`kind`)

Сейчас `LayoutNode.type` — это рендер-тип (`service | custom | group`). Нужно второе поле — **что** это за компонент.

- **Что сделать:**
  - Добавить поле `kind` в `LayoutNode` (фронт: `api.ts`, бэкенд: `layout.py`).
  - `type` остаётся для рендеринга (`service`, `component`, `group`), `kind` — семантика.
  - Начальный набор kind: `user`, `vpn-gateway`, `network-segment`, `vpc`, `interconnect`, `load-balancer`, `cdn`, `waf`, `api-gateway`, `k8s-cluster`, `service`, `managed-db`, `managed-cache`, `object-storage`, `dns`, `custom`.
  - Миграция: существующие ноды без `kind` → `kind: "service"` (для type=service) или `kind: "custom"` (для type=custom).
  - Бэкенд `layout.py` сохраняет и читает `kind` как есть (без валидации — список расширяемый).
- **Готово когда:** поле `kind` сохраняется в JSON-раскладке; фронт читает его; старые раскладки без `kind` работают без ошибок (fallback).

При создании ноды (ПКМ / плавающая панель) пользователь выбирает **тип компонента** из каталога вместо голого «custom».

- **Что сделать:**

  - В контекстном меню / плавающей панели — группированный список: «Актор» (user, bot), «Сеть» (vpn-gateway, interconnect, vpc), «Точка входа» (load-balancer, cdn, waf), «Кластер» (k8s-cluster), «Сервис», «Managed-ресурс» (managed-db, managed-cache, object-storage, dns).
  - При выборе пресета нода создаётся с правильной иконкой, цветом и `kind`.
  - Иконки: Lucide уже есть в проекте — `Globe`, `Shield`, `Server`, `Database`, `Container`, `Network`, `Cloud`, `HardDrive` и т.д.
- **Готово когда:** пользователь может создать ноду «VPN Gateway» или «Managed DB» в два клика; нода визуально отличается от обычного сервиса; `kind` сохраняется в раскладке.

### [ ] Фаза 4: Привязка blackbox-проб к любому типу ноды

Сейчас мониторинг привязан только к нодам типа `service` (через `matchServiceId`). Нужно чтобы **любой** компонент мог показывать статус пробы.

### [ ] Фаза 5: Managed-ресурсы из облачного API (второй datasource)

- **Что сделать:**
  - Модель второго datasource: `{ type: "cloud-api", provider: "yandex" | "aws" | "gcp", endpoint, credentials_env }`.
  - Бэкенд: адаптер для облачного API (начать с одного провайдера, например Yandex Cloud — `compute.api.cloud.yandex.net`).
  - Получать статус: managed DB (running/stopped/error), LB (healthy/unhealthy), Object Storage (available).

  - UI: в панели ноды — статус из облачного API рядом с blackbox-пробами.

## Проекты и хранение

### [ ] Soft delete проектов + корзина + восстановление

- **Готово когда:** `DELETE` не стирает запись безвозвратно; есть список «удалённые»; восстановление доступно согласно матрице ролей (до внедрения ролей — только «не показывать в основном списке» + восстановление для всех с доступом к API, если иначе не оговорено).
- **Зависимость:** логично после матрицы ролей или явное правило «пока все админы».

### [ ] Import / export проекта

- **Готово когда:** один файл (например JSON) содержит имя, фильтры, раскладку, `service_configs` по необходимости; import создаёт новый проект или перезаписывает с подтверждением; экспорт совпадает с импортом (round-trip).

### [ ] S3 (или S3-совместимое API) для артефактов

- **Готово когда:** конфиг: endpoint, bucket, префикс, credentials через env или секреты; чтение/запись раскладок и/или проектов; fallback или миграция с `data/` описаны в комментарии к настройке или коротком README-фрагменте в репо.

## Редактор карты

### [x] Undo / redo (~10 шагов в каждую сторону)

- **Входит в шаг:** перемещение узлов, добавление/удаление узла и ребра, изменения group (если есть в стеке), правки подписей на карте — по согласованию с первой реализацией (список зафиксировать в PR).
- **Не обязательно в первой версии:** undo после автосохранения на сервер (достаточно локального стека до следующей полной перезагрузки страницы).
- **Готово когда:** Ctrl+Z / Ctrl+Shift+Z или кнопки; глубина ≤10; не ломает сохранение раскладки.
- **Статус:** сделано — история (ноды+рёбра+service_configs), Ctrl/Cmd+Z и Ctrl/Cmd+Shift+Z, кнопки Undo/Redo в плавающей панели карты, глубина 10.

### [ ] Стрелка как отдельный объект с произвольными концами

- **Связано с:** «Модель сетевого пути, Фаза 3» — рёбра с метаданными. Эта задача про **аннотационные** стрелки (свободные, не привязанные к нодам); Фаза 3 — про **рёбра** между нодами с протоколом/описанием. Можно делать параллельно.
- **Готово когда:** на карте можно добавить объект «стрелка»/аннотация; оба конца позиционируются независимо (якоря в координатах или привязка к сторонам узла — зафиксировать в UI); сохраняется в JSON раскладки; не путается с обычным ребром зависимости.

## Мониторинг

### [x] Blackbox-экспортёры на карте + «игнор источника»

- **Готово когда:** видно, какие экспортёры участвуют в пробах сервиса (список на карте или в панели сервиса); у каждого можно пометить «не учитывать»; агрегат ok/down и счётчики исключают отключённые источники; настройка сохраняется в раскладке или `service_configs`.
- **Статус:** начато — добавлены переключатели «игнорировать источник» в панели сервиса; состояние сохраняется в `service_configs` и исключается из агрегации (цвет / счётчики / expected). Точки blackbox на карточке тоже скрывают игнорированные источники.

### [ ] Blackbox-пробы для сетевых хопов (Alloy/Prometheus)

Для покрытия цепочки User → ... → Service нужно добавить probe targets на каждый хоп. Это работа **вне** кодовой базы ProbeMap, но результат видим на карте.

- **Что сделать (в homelab-gitops / Alloy конфиге):**
  - VPN gateway: ICMP ping или TCP probe на endpoint.
  - Interconnect: ICMP через туннель до remote endpoint.
  - Load balancer: HTTP probe на VIP / health-check URL.
  - Managed DB: TCP probe на порт (5432, 3306, 6379).
  - Object storage: HTTP HEAD на endpoint.
  - Каждому target — лейбл `service` (или маппинг из label_map), чтобы ProbeMap подхватил автоматически.
- **Готово когда:** в VictoriaMetrics есть серии `probe_success` для ≥3 типов хопов (например: vpn, lb, managed-db); ProbeMap показывает их как сервисы в каталоге; их можно привязать к нодам на карте.
- **Примечание:** ProbeMap уже умеет показывать эти данные. Задача — правильно настроить scrape targets. Это самый быстрый способ получить мониторинг всей цепочки без нового кода.
