## Концепция

ProbeMap тащит из Prometheus-совместимого датасорса метрику `probe_success` и пририсовывает статусы объектам на карте.

### Сущности на карте

| Сущность | Примеры | Пробы | kind в nodeKinds |
|----------|---------|-------|------------------|
| **Сервис** | API, фронтенд, бэкенд | HTTP, TCP, UDP | `service` |
| **Ресурс** | ВМ, LB, managed DB, кеш, DNS | ICMP (+ TCP для managed) | `managed-db`, `managed-cache`, `load-balancer`, `dns`, ... |
| **Кластер** | K8s-кластер, группа ВМ | Агрегат по лейблу | `k8s-cluster` (+ новые) |
| **Допобъект** | Пользователь, роутер, VPC, VPN | Нет (адрес может быть неизвестен) | `user`, `vpn-gateway`, `vpc`, `interconnect`, ... |
| **Область** | Визуальная группировка | Нет | `type: "group"` (React Flow) |

Когда ProbeMap сконфигурирован на датасорс и (опционально) метрики отфильтрованы по лейблам — мы получаем общий каталог доступных объектов. Дальше размещаем их на карте и связываем как хотим.

## Блок A — Модель сущностей (ядро)

> Цель: карта различает сервисы, ресурсы, кластеры и допобъекты визуально и по поведению.

### [ ] A1. Объединение `service` / `custom` в единый рендер-тип

Сейчас в React Flow два типа — `type: "service"` и `type: "custom"`. С появлением `kind` это разделение бессмысленно: любая нода может иметь привязку к метрикам.

- **Что сделать:**
  - Один `type: "service"` (или `"component"`) для всех не-group нод.
  - Визуальные различия — по `kind` и наличию привязки, не по `type`.
  - Миграция: `type: "custom"` → единый тип при чтении раскладки.
- **Готово когда:** один React-компонент рендерит все не-group ноды; старые раскладки работают без потерь.

### [ ] A2. Визуальное разделение по типу сущности

Сейчас все ноды выглядят одинаково (одна карточка). Нужно визуально отличать разные `kind`.

- **Что сделать:**
  - **Сервисы** (`kind: "service"`): текущий стиль — полная карточка с портами, HTTP/TCP бейджами.
  - **Ресурсы** (`kind: managed-*`, `load-balancer`, ...): компактнее, акцент на ICMP-статусе (зелёный/красный индикатор), без развёрнутого списка портов по умолчанию.
  - **Допобъекты** (`kind: user`, `vpn-gateway`, `vpc`, ...): иконка + подпись, без панели мониторинга (если нет привязки). Серый/нейтральный стиль.
  - **Кластеры** (`kind: k8s-cluster`, ...): визуально как контейнер или крупный узел (не group, но акцент на агрегированном статусе).
  - Стиль определяется по `group` из `nodeKinds.ts` (actor / network / entry / cluster / service / managed / other).
- **Готово когда:** глядя на карту, можно отличить сервис от ресурса от допобъекта без чтения подписей.

### [ ] A3. Допобъекты без адреса

Сейчас нода без привязки к метрикам показывает `unknown` статус. Для user/router/VPC это нормальное состояние, не ошибка.

- **Что сделать:**
  - Для `kind` из групп `actor` / `network` / `other` (без `matchServiceId`) — не показывать статус вообще: нет бейджа, нет пульсации, нет рамки unknown.
  - Явный UI-индикатор: «без мониторинга» (серая точка или отсутствие точки) vs «мониторинг подключен» (зелёная/красная).
  - Панель «МОНИТОРИНГ» по-прежнему доступна — можно привязать, если адрес известен.
- **Готово когда:** User на карте — просто иконка с подписью, без «unknown» статуса; VPN-шлюз с привязкой показывает ICMP-статус.

### [ ] A4. Кластеры — агрегация по лейблу

Сейчас `k8s-cluster` — просто иконка. Нужна возможность объединять ресурсы в кластер по лейблу из метрик.

- **Что сделать:**
  - Нода `kind: k8s-cluster` (и новые: `vm-cluster`, `server-group`) может иметь поле `clusterLabel` + `clusterValue` — фильтр по лейблу из `probe_success`.
  - Агрегированный статус кластера: ok (все ресурсы ok), warn (часть down), down (все down).
  - В панели кластера — список входящих ресурсов с их статусами.
  - Ресурсы кластера могут быть и отдельными нодами на карте (двойная визуализация: внутри кластера + отдельно).
- **Зависимость:** A1 (единый тип ноды).
- **Готово когда:** K8s-кластер на карте показывает агрегат по лейблу; раскрыв панель, видно список ресурсов и их статусы.

## Блок B — Каталог и палитра

> Цель: палитра показывает все доступные объекты, сгруппированные по типу, и позволяет быстро добавлять на карту.

### [ ] B1. Группировка каталога по kind

Сейчас палитра — плоский список сервисов из `probe_success`. Нужно группировать по типу пробы/модулю.

- **Что сделать:**
  - Бэкенд: в ответе `/api/projects/{id}/services` для каждого сервиса добавить поле `probe_kind` — автоопределение по `module` лейблу: `http*` → service, `icmp*` → resource, `tcp*`/`udp*` → resource или service (по эвристике или маппингу).
  - Фронт: палитра группирует по `probe_kind` (или по `kind` если нода уже на карте).
  - Поиск по палитре ищет по всем группам.
- **Готово когда:** палитра показывает «Сервисы (HTTP)», «Ресурсы (ICMP)» как секции; поиск работает сквозь секции.

### [ ] B2. Автоматический `kind` при добавлении из каталога

При добавлении сервиса из палитры на карту — подставлять `kind` по типу проб.

- **Что сделать:**
  - HTTP-пробы → `kind: "service"`.
  - ICMP-пробы → `kind: "managed-db"` / `"load-balancer"` / ... (по эвристике из имени или лейблов, fallback — `"custom"`).
  - Пользователь может поменять `kind` после добавления.
- **Зависимость:** B1.
- **Готово когда:** ICMP-проба `managed-postgres` добавляется с `kind: "managed-db"` и соответствующей иконкой.
